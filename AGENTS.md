# AGENTS.md

This repository (`rbac-config`) holds the declarative JSON configurations for roles and permissions that the Red Hat Hybrid Cloud Console RBAC service seeds into every tenant. It also contains Kessel Schema Language (KSL) files that compile to SpiceDB authorization schemas for the V2 authorization model. There is no application code here -- only JSON configs, KSL schemas, and CI automation.

## Repository Layout

```
configs/
  stage/                    # Stage environment (deploys Tuesdays)
  prod/                     # Production environment (deploys Thursdays)
    permissions/<app>.json  # One file per app; filename = app namespace
    roles/<app>.json        # One file per app; contains a "roles" array
    schemas/
      src/*.ksl             # KSL source files for Kessel/SpiceDB
      migrated_apps.lst     # Apps using V2 authorization
      schema.zed            # GENERATED -- never edit manually
schemas/
  permissions.schema        # JSON Schema for permission files
  roles.schema              # JSON Schema for role files
_private/                   # Generated artifacts -- never edit
```

## Cross-Cutting Conventions

### Stage-First, Always

All changes go to `configs/stage/` first. Promote to `configs/prod/` only after stage validation. Stage and prod can intentionally diverge (different versions, different `migrated_apps.lst` entries, stage-only KSL files like `subscriptions.ksl`). Never blindly copy stage files to prod.

### File Naming Determines Identity

The permission filename (minus `.json`) becomes the application namespace in the `app:resource:verb` triple. `cost-management.json` means all its permissions start with `cost-management:`. Use lowercase with hyphens for multi-word app names. The role filename does not need to match the permission filename -- `insights.json` (roles) references `advisor:` permissions.

### Version Bumping Is Mandatory

The `version` integer field is the only mechanism that triggers re-seeding in the RBAC service. If you change anything in a role and do not increment `version`, the change will be silently ignored in production. New roles must start at version 2 or higher.

### The `name` Field Is Immutable

The `name` field is the database primary key. Changing it creates a duplicate role -- the old one persists forever. Use `display_name` to change what users see in the UI.

### Permission and Role Files Are Not 1:1

Some apps have permissions but no roles (e.g., `advisor.json`, `playbook-dispatcher.json`) -- their permissions are consumed by roles in other files. Some role files reference permissions from many apps (e.g., `rhel.json` aggregates permissions from advisor, compliance, inventory, patch, vulnerability, and more). This cross-app referencing is intentional and common.

### JSON Formatting

Use 4-space indentation in permission files (the existing convention) and 2-space indentation in role files. Include trailing newlines. No trailing commas. Maintain consistent ordering: `name`, `display_name`, `description`, `system`, `platform_default`, `admin_default`, `version`, then `access` or `external`.

### KSL Naming Conventions

In KSL files, hyphens in app/resource names are replaced with underscores (e.g., `malware-detection` becomes `malware_detection`). V2 permission names follow the pattern `{app}_{resource}_{v2_verb}` (e.g., `advisor_recommendation_results_view`). Host-scoped permissions use the `_assigned` suffix for the intermediate relation before the contingent intersection.

## Files You Must Never Edit Manually

- `_private/configmaps/` -- Generated by CI on merge to master.
- `configs/*/schemas/schema.zed` -- Generated from KSL source files. Edit `.ksl` files instead.
- `configs/*/schemas/src/rbac_v1_permissions.json` -- Build artifact (gitignored).

## Common Pitfalls

1. **Forgot to bump `version`** -- The single most common mistake. Changes will not take effect.
2. **Renamed `name` instead of setting `display_name`** -- Creates a ghost duplicate role in the database.
3. **Deleted a role file expecting removal** -- Role deletion is not automatic; contact the RBAC team.
4. **Added permissions to roles but not to `configs/*/permissions/`** -- The permission works initially but is a candidate for future removal.
5. **Changed only one environment** -- If the change applies to both, update both `stage/` and `prod/`. Check whether `migrated_apps.lst` needs updating too.
6. **Edited `schema.zed` directly** -- It will be overwritten. Edit the `.ksl` source files.
7. **Used `make ksl-schema-*` instead of `make ksl-test-schema-*` locally** -- The non-test targets overwrite committed files. Use the test targets which write to `_private/test-schema/`.
8. **Wildcard `"*"` resource missing from permissions file** -- Most apps need `"*": [{"verb": "*"}]` for administrator roles that reference `app:*:*`.

## PR and Deployment Workflow

1. Open a PR against `master` with changes in `configs/`.
2. CI validates JSON schemas, permission dependencies, and SpiceDB schemas for both environments.
3. After merge, an automated PR generates ConfigMaps on the `configmaps-schema` branch.
4. That automated PR must also be merged.
5. A separate MR in `app-interface` (GitLab) bumps the `ref` to trigger actual deployment.

Reviews and deployments happen on the same day: stage on Tuesdays, prod on Thursdays.
