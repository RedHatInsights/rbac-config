## Claude Code-Specific Workflow

### Pre-PR Validation Checklist

Before creating a pull request, Claude Code should:

1. **Run schema validation locally** for any modified environment:
   ```sh
   make ksl-test-schema-stage   # if stage configs changed
   make ksl-test-schema-prod    # if prod configs changed
   ```

2. **Verify critical constraints**:
   - Role `version` incremented for any modified roles
   - Permission filename matches app namespace in role `access` arrays
   - Both stage and prod schemas validate if KSL files were modified

3. **Never run** the non-test schema targets (`make ksl-schema-stage`, `make ksl-schema-prod`) -- these overwrite committed files

### Build and Test Commands

```sh
make init                    # Install ksl and generate-v1-only-permissions (requires Go)
make check-go-tools          # Verify required tools are installed
make ksl-test-schema-stage   # Build and validate stage schema → _private/test-schema/
make ksl-test-schema-prod    # Build and validate prod schema → _private/test-schema/
```

**Do not use**: `make ksl-schema-stage` or `make ksl-schema-prod` for local testing (they overwrite committed `schema.zed` files).

### CI Pipeline Awareness

The PR workflow (`.github/workflows/pr.yml`) runs five sequential validation steps. All must pass:

1. JSON Schema validation -- Permissions (`schemas/permissions.schema`)
2. JSON Schema validation -- Roles (`schemas/roles.schema`, non-strict)
3. Permission dependency validation (`requires` field integrity)
4. V1-only permissions generation + KSL schema compilation
5. SpiceDB schema validation (both stage and prod)

After merge to `master`, an automated PR generates ConfigMaps on the `configmaps-schema` branch. That PR must also be merged.

### Behavioral Preferences

- **Always validate schemas locally** before suggesting a PR (run `make ksl-test-schema-*` for modified environments)
- **Check both environments** when modifying KSL files or `.lst` files -- stage and prod schemas are validated independently in CI
- **Never commit generated artifacts**: `schema.zed` is regenerated by CI; `rbac_v1_permissions.json` is gitignored
- **Increment role versions**: If you modify a role's `access`, `display_name`, or flags, bump `version` or the change will be ignored in production
- **Use test targets only**: Local schema builds should always use `make ksl-test-schema-*` which write to `_private/test-schema/` instead of overwriting committed files

### File Edit Patterns

When modifying configs:

1. **Permission files** (`configs/*/permissions/*.json`): Use 4-space indentation
2. **Role files** (`configs/*/roles/*.json`): Use 2-space indentation
3. **KSL files** (`configs/*/schemas/src/*.ksl`): Replace hyphens with underscores in app/resource names
4. **Never edit**: `schema.zed`, `rbac_v1_permissions.json`, or anything under `_private/configmaps/`

### Deployment Context

- **Stage**: Promoted Tuesdays (review same day)
- **Prod**: Promoted Thursdays (review same day)
- Stage-first rule: All changes go to `configs/stage/` first, then promote to `configs/prod/` after validation
- Stage and prod configs can intentionally diverge (different KSL files, different `migrated_apps.lst`)
