# Testing Guidelines

## CI Validation Pipeline (PR Workflow)

Every pull request to `master` must pass all five validation steps defined in `.github/workflows/pr.yml`. These run sequentially in a single job and all must pass before merge.

### 1. JSON Schema Validation -- Permissions

All files matching `configs/*/permissions/*.json` are validated against `schemas/permissions.schema`. Key constraints:

- Top-level keys are resource names (arbitrary strings)
- Values are arrays of permission objects, each requiring a `verb` field
- Allowed verbs: `*`, `read`, `write`, `create`, `update`, `upload`, `delete`, `list`, `link`, `unlink`, `order`, `execute`, `view`, `grant`, `revoke`
- Optional fields: `description` (string), `requires` (array of verb strings)
- No additional properties allowed on permission objects

### 2. JSON Schema Validation -- Roles

All files matching `configs/*/roles/*.json` are validated against `schemas/roles.schema` (non-strict mode). Key constraints:

- Root object must have a `roles` array
- Each role requires: `name`, `description`, `system`, `version`
- Each role must have exactly one of: `access` (array of permissions) or `external` (object with `id` and `tenant`)
- Permission strings in `access` must match pattern: `<app>:<resource>:<verb>` where verb is from the allowed verb list
- `resourceDefinitions` use `attributeFilter` with `key`, `operation` (`equal` or `in`), and `value`
- Optional fields: `display_name`, `platform_default`, `admin_default`

### 3. Permission Dependency Validation

Uses `RedHatInsights/rbac-config-actions/validate-permission-dependencies` against `configs/*/permissions/*.json`. This validates that any `requires` array entries reference verbs that actually exist for the same app/resource combination. For example, if `create` requires `["read"]`, then `read` must be defined in the same resource array.

### 4. KSL-to-SpiceDB Schema Generation

The pipeline generates V1-only permissions data and then compiles KSL schemas for both environments:

- **V1 permissions generation**: Reads `configs/{stage,prod}/permissions/*.json` and `configs/{stage,prod}/schemas/*.lst` files to produce `configs/{stage,prod}/schemas/src/rbac_v1_permissions.json` (gitignored build artifact)
- **Schema compilation**: Uses `RedHatInsights/rbac-config-actions/validate-schema` to compile `.ksl` source files plus the generated JSON into `configs/{stage,prod}/schemas/schema.zed`

The `.lst` files (`migrated_apps.lst`, `hostsonly_apps.lst`) control which apps get V1-only permission entries in the generated schema.

### 5. SpiceDB Schema Validation

Uses `authzed/action-spicedb-validate` to validate both generated `schema.zed` files are syntactically and semantically valid SpiceDB schemas. Runs separately for stage and prod.

## Local Development

### Prerequisites

- Go (any recent version should work)
- Run `make init` to install required tools (`ksl` and `generate-v1-only-permissions`)
- Run `make check-go-tools` to verify installation

### Building Test Schemas Locally

Use the test targets which write to `_private/test-schema/` (gitignored) instead of overwriting the committed `schema.zed` files:

```sh
make ksl-test-schema-stage   # writes _private/test-schema/stage-schema.zed
make ksl-test-schema-prod    # writes _private/test-schema/prod-schema.zed
```

**Do not use** `make ksl-schema-stage` or `make ksl-schema-prod` for local testing -- these overwrite the committed `configs/{env}/schemas/schema.zed` files, which are regenerated by the automated master workflow.

### Build Dependency Chain

The Makefile encodes a dependency chain for production schema targets:

1. `configs/{env}/schemas/src/rbac_v1_permissions.json` depends on `configs/{env}/permissions/*.json` + `configs/{env}/schemas/*.lst`
2. `configs/{env}/schemas/schema.zed` depends on `configs/{env}/schemas/src/*.ksl` + the generated JSON

The test schema targets (`ksl-test-schema-*`) are marked as `.PHONY` targets, which means they always run when invoked regardless of whether their dependencies have changed.

## Repository Structure for Testing

```
configs/
  stage/                    # Stage environment (deployed Tuesdays)
    permissions/*.json      # Permission definitions per app
    roles/*.json            # Role definitions per app
    schemas/
      src/*.ksl             # KSL schema source files
      *.lst                 # App migration lists (migrated_apps.lst, hostsonly_apps.lst)
      schema.zed            # Generated SpiceDB schema (DO NOT edit manually)
  prod/                     # Production environment (deployed Thursdays)
    (same structure as stage)
schemas/
  permissions.schema        # JSON Schema for permission files
  roles.schema              # JSON Schema for role files
_private/
  configmaps/               # Generated ConfigMaps (automated PR)
  test-schema/              # Local test output (gitignored)
```

## Key Rules for Making Changes

1. **Never edit `schema.zed` directly** -- it is generated. Edit `.ksl` files in `configs/{env}/schemas/src/` instead.
2. **Never commit `rbac_v1_permissions.json`** -- it is gitignored and generated during CI/build.
3. **Stage and prod are independent** -- changes can differ between environments. Stage has `subscriptions.ksl` which prod does not; the `.lst` files may differ.
4. **Permission file names are app names** -- the filename (minus `.json`) becomes the app namespace in permission strings (e.g., `rbac.json` defines permissions for `rbac:*:*`).
5. **Role version must be incremented** when updating a role, or the RBAC service will not pick up the change.
6. **Both environments must pass** -- CI validates stage and prod independently. A KSL syntax error in only one environment will still block the PR.
7. **`requires` dependencies are intra-resource only** -- a verb can only require other verbs defined on the same resource within the same app file.

## Master Branch Automation

After merge to `master`, `.github/workflows/master.yml` runs the full schema generation pipeline and creates an automated PR to the `configmaps-schema` branch with:

- Regenerated `schema.zed` for both environments
- Converted ConfigMaps in `_private/configmaps/`

This automated PR must also be merged for changes to reach deployment.

## Validation Checklist

Before submitting a PR, verify locally:

- [ ] Permission JSON files are valid against `schemas/permissions.schema`
- [ ] Role JSON files are valid against `schemas/roles.schema`
- [ ] Any `requires` arrays reference verbs that exist on the same resource
- [ ] `make ksl-test-schema-stage` succeeds (if stage schemas were changed)
- [ ] `make ksl-test-schema-prod` succeeds (if prod schemas were changed)
- [ ] Role `version` field is incremented for any modified roles
